# 星筑科技官网项目规则

本项目是星筑科技官方网站项目，定位是长期维护的企业官网工程，不是一次性生成的静态页面作品。

## 项目信息

- 公司名称：星筑科技
- 项目类型：企业官网
- 技术栈：React + Vite + Tailwind CSS + Framer Motion + Lucide React
- 主要目标：高级、克制、科技感、可维护、可扩展

## 长期架构原则

1. 不要从零重做整个官网，除非用户明确要求。
2. 不要删除现有页面内容，除非用户明确要求。
3. 不要把项目改成五个互不相关的 HTML 页面。
4. 保持五个页面在同一个 React/Vite 官网系统中。
5. 保持统一导航栏、统一页脚、统一视觉系统。
6. 所有代码改动完成后，必须保证 `npm run build` 可以通过。

## 目录职责

- `src/App.jsx`：负责全站路由。
- `src/main.jsx`：负责 React 启动入口。
- `src/shared`：负责全站公共组件、公共布局、公共数据、公共动画。
- `src/pages/Home`：只负责首页。
- `src/pages/About`：只负责关于页面。
- `src/pages/Solutions`：只负责解决方案页面。
- `src/pages/Insights`：只负责洞察页面。
- `src/pages/Contact`：只负责联系页面。

## 页面文件规则

每个页面目录应尽量保持以下结构：

- 页面入口文件：例如 `Home.jsx`
- 页面数据文件：例如 `home.data.js`
- 页面配置文件：例如 `home.config.js`
- 页面动画文件：例如 `home.motion.js`
- 页面板块目录：`sections/`
- 页面说明文件：`README.md`

页面上展示的标题、描述、按钮文字、卡片内容、文章列表、标签、统计数字等，优先放到对应页面的 `*.data.js` 文件。

组件文件负责展示结构，不应堆积大量固定文案。

## 公共组件规则

如果多个页面出现重复结构，应优先抽到 `src/shared/components`。

适合抽为公共组件的内容包括：

- 按钮
- 卡片
- 页面标题区
- CTA 区域
- 标签
- 统计数字卡片
- 文章卡片
- 表单字段

公共组件只负责通用结构和样式，不写具体业务文案。

## 设计规则

1. 视觉风格要高级、克制、科技感。
2. 避免廉价模板感。
3. 避免过度花哨的装饰。
4. 动效要服务内容，不要喧宾夺主。
5. 页面要适配桌面端和移动端。
6. 不要让文字、按钮、卡片在小屏幕上重叠或溢出。

## Agent Team 使用规则

当用户要求多页面维护、架构整理、五个页面协同修改，或明确提到 Agent Team 时，优先使用 `website-agent-team` skill。

不要把 Agent Team 的完整流程写在普通对话里重复执行，应读取并遵守该 skill。

## Bug 审查与系统化排查规则

当用户要求代码 bug 审查、页面异常排查、构建失败修复、路由错误、组件报错、样式错乱、交互失效，或 `npm run build` 失败时，优先使用 `website-bug-audit` skill。

处理 bug 时不要猜测式修改。必须先完成以下步骤：

1. 读取完整错误信息或复现异常现象。
2. 定位异常发生在哪一层：路由、页面组件、shared 公共组件、数据文件、样式、动画、构建配置或依赖。
3. 对照项目中正常工作的相似页面或组件。
4. 明确根因后再做最小范围修复。
5. 修复后重新运行验证命令，至少包括 `npm run build`。

## 完成标准

每次完成官网代码修改后，应确认：

1. 修改了哪些文件。
2. 是否影响路由。
3. 是否影响 shared 公共组件。
4. 是否影响五个页面互相跳转。
5. `npm run build` 是否通过。
